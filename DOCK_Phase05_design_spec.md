## 1. 한 줄 요약

GT Mechanik(gt-mechanik.com)의 가변폰트 타입 테스터 UI 패턴을 차용한, **사용자가 자기 발화의 시각적 톤을 직접 결정하는 모바일 화면**.

---

## 2. 제품 맥락

**DOCK이란?**
한국 대학교 캠퍼스에 설치되는 공공 발화 시스템. 외벽 프로젝션(4×6m, 11개 메시지가 아래→위로 떠오르는 풍경) + R2 이동식 본체(NFC 거치대)로 구성. 학생이 R2 후면에 폰을 안착시키면 웹앱이 자동 실행되고, 60자 메시지를 작성·톤 결정·도킹하면 외벽에 7일간 메시지가 표시됨.

**서영(페르소나)이 해결하는 5가지 벽:**
- 형식의 벽 (다른 매체와 같은 형식·톤 강요)
- 위계의 벽 (학생회 허가 등 절차)
- 시선의 벽 (비익명, 학과 인간관계 부담)
- 편집권의 벽 (본인 서사의 형식을 결정할 수 없음)
- 시한의 벽 (한 시간 후 묻힘)

**이 화면이 해결하는 것: 편집권의 벽.**
사용자가 본인 발화의 시각적 형식(폰트·굵기·기울기·색상 등)을 의도적으로 결정하는 단계. AI 추천 없음 — 발화자의 형식 결정권이 절대 원칙.

**9 Phase 사용 흐름 중 Phase 05의 위치:**
1. 발견 → 2. 접근 → 3. 접속(NFC 도킹) → 4. 작성(60자 텍스트) → **5. 톤 결정 (← 이 화면)** → 6. 도킹(맡기기) → 7. 마주봄 → 8. 합류 → 9. 떠남

## 3. 핵심 디자인 원칙

1. **모바일 우선** — 폰을 R2 거치대에 안착시킨 채로 조작. 한 손 조작 가능해야 함.
2. **한글 우선** — 모든 텍스트와 폰트는 한글이 기본. 컨트롤 라벨만 영문 약어(GT Mechanik 패턴).
3. **만지작거리기로 발견** — 사용자는 추상 감정 슬라이더가 아니라 구체적 타이포그래피 변수를 만지며 본인 톤을 *발견*함.
4. **AI 추천 금지** — 형식 결정권은 발화자에게. 알고리즘이 "이 톤 추천"하지 않음.
5. **STAR와 COLOR는 우연성** — 두 가지만 무작위로 작동 (그래픽·색상). 다른 모든 컨트롤은 발화자의 명시적 결정.
6. **줄 펄스는 베이스라인** — 외벽 디스플레이의 "호흡"이 미리보기에서도 항상 작동. 사용자가 끄거나 조절할 수 없음 (시스템이 보장하는 외벽 운동성).

## 5. 6개 컨트롤 + 3개 액션 상세

### 5.1. FONT (폰트 패밀리) — 3개 버튼

| 라벨 | 폰트 | CSS font-family |
|:--|:--|:--|
| 모노 | D2Coding | `"D2Coding", "Courier New", monospace` |
| 고딕 | Pretendard Variable | `"Pretendard Variable", -apple-system, sans-serif` |
| 명조 | Noto Serif KR Variable | `"Noto Serif KR Variable", serif` |

**기본값:** 고딕

**의미:** 폰트 패밀리는 발화의 가장 근본적인 음색을 결정함. 모노 = 기계적·또박또박, 고딕 = 일상적·명료, 명조 = 사색적·서간체.

---

### 5.2. TONE (장체↔평체) — 3개 버튼

| 라벨 | 값 (scaleX) |
|:--|:--|
| 장체 | 0.7 |
| 평 | 1.0 |
| 평체 | 1.3 |

**기본값:** 평 (1.0)

**구현:** `transform: scaleX(N)` — 진짜 wdth 가변 축이 한글 가변폰트에 없으므로 CSS transform으로 흉내.

**의미:** 글자의 가로 비율. 장체는 좁고 키 큰 발화감, 평체는 넓고 여유로운 발화감.

---

### 5.3. WGHT (굵기) — 6개 버튼

| 라벨 | 값 (font-weight) |
|:--|:--|
| THIN | 200 |
| LGHT | 300 |
| RGLR | 400 (기본) |
| SEMI | 600 |
| BOLD | 800 |
| BLAK | 900 |

**구현:** `font-weight` + `font-variation-settings: "wght" N` 동시 적용. 가변폰트가 자동 보간.

**의미:** 발화의 무게감·단호함.

**줄 펄스와의 상호작용 (중요):** 사용자가 선택한 WGHT는 줄 펄스의 **최고점**으로 작동함 (5.9 참조). 베이스 가중치는 자동 계산됨 (`Math.max(100, Math.round(userWght * 0.5))`). 즉:
- THIN(200) → 100 ↔ 200 (미세 펄스)
- RGLR(400) → 200 ↔ 400 (보통 펄스)
- BOLD(800) → 400 ↔ 800 (강한 펄스)
- BLAK(900) → 450 ↔ 900 (최강 펄스)

사용자가 발화의 *강도*를 결정하면, 줄 펄스의 *진폭*이 그에 비례.

---

### 5.4. SLNT (기울기) — 2개 버튼

| 라벨 | 값 (skewX) |
|:--|:--|
| UPRT | 0deg (기본) |
| OBLQ | -8deg |

**구현:** `transform: skewX(N)` — 한글 가변폰트엔 slnt 축이 없어서 CSS transform으로 흉내. -8도가 자형 일그러짐의 한계점.

**의미:** 발화의 급함·운동성.

**알려진 한계:** -8도에서 한글 자모가 살짝 일그러져 보임. 이건 한글 가변폰트의 현실적 한계로 받아들임 (또는 추후 본인 한글 가변폰트 제작 시 진짜 slnt 축으로 교체).

### 5.5. SIZE (크기) — 슬라이더

- 범위: 18 ~ 56 (px)
- step: 1
- 기본값: 32

### 5.6. STAR (그래픽 오버레이) — 액션 버튼

**동작:**
- 누를 때마다 5종 그래픽 중 랜덤 1개가 미리보기 뒤쪽에 오버레이됨
- 이미 그래픽이 떠 있는 상태에서 다시 누르면: 다른 그래픽으로 바뀜 (85% 확률) 또는 끄기 (15% 확률)
- 그래픽은 메시지 텍스트 *뒤에* 위치 (z-index 1, 텍스트는 z-index 2)

**5종 그래픽 + 모션:**

| 그래픽 | 모션 |
|:--|:--|
| 구체 (sphere) | 14초 주기로 천천히 회전, 위도/경도 격자선 |
| 폴카닷 (dots) | 3.2초 주기 펄스 (opacity 0.5↔1.0) |
| 격자 (grid) | dashed 세로선이 흐르듯 stroke-dashoffset 애니메이션 |
| 동심원 (concentric) | 5개 원이 4.5초 주기로 호흡 (scale 0.92↔1.06) |
| 파동 (waves) | 6개 sine 곡선이 좌→우로 슬라이드 |

**색상:** 그래픽 색은 현재 팔레트의 `graphic` 색상을 따름 (COLOR 버튼이 바꿈).

### 5.9. 줄 강조 펄스 (베이스라인 모션, 컨트롤 없음)

**무엇:** 미리보기 텍스트가 줄 단위로 굵어졌다 돌아오는 사이클 모션. 한 줄씩 강조됨.

**왜:** 외벽 디스플레이의 "호흡". 메시지가 정지된 텍스트가 아니라 *누군가 읽고 있는 듯한 라이브감*을 만듦. 11개 메시지가 각자 자기 박자로 줄을 강조하면 풍경 전체가 합창처럼 보임. "사적 발화가 공적 풍경으로 흡수되는 의례"라는 DOCK의 주제 강화.

**작동 모델 — Option B (사용자 WGHT가 펄스 최고점):**
- 베이스 가중치: `max(100, round(userWght * 0.5))`
- 펄스 최고점: `userWght`
- 사용자가 WGHT 컨트롤로 정한 값 = 펄스의 최고점
- 펄스 진폭은 사용자의 WGHT 선택에 비례 (THIN이면 거의 안 보이고, BLAK이면 강함)
- 의미: 발화자가 자기 발화의 *강도*를 결정 → 외벽의 호흡 진폭도 그에 비례

**타이밍:**
- 한 줄당 액티브 시간: 2.4초
- 트랜지션: font-weight 0.9초 ease-in-out (부드러운 가중치 변화)
- 펄스 사이클: `(줄 수) × 2.4초`
  - 2줄 메시지: 4.8초 전체 사이클
  - 3줄 메시지: 7.2초 전체 사이클

**컨트롤 없음:** 사용자가 켜고/끄거나 속도를 조절할 수 없음. 베이스라인으로 항상 작동.

**예외 상태:**
- 사용자가 미리보기 텍스트를 *편집 중*(focus): 펄스 일시 정지, 전체 텍스트가 단일 가중치(userWght). 편집 끝나면(blur) 펄스 재개.
- 빈 텍스트: 펄스 중단.
- 한 줄 메시지: 한 줄에서만 깜박이는 펄스 (의도된 동작).

**구현 노트:**
- 단어 단위 `<span>`으로 텍스트 래핑
- `getBoundingClientRect().top`으로 시각적 줄 그룹 감지
- 각 단어 span에 `data-line="N"` 속성 부여
- 동일한 line 번호의 단어들이 동시에 active 클래스 토글
- CSS 변수로 `--wght-base`, `--wght-active` 관리
- `transition: font-weight 0.9s ease-in-out` 적용

**외벽으로의 확장 (Phase 06+ 이후):**
- 외벽의 11개 메시지 각각이 자기만의 펄스를 가짐
- 동기화하지 않음 — 각 메시지마다 시작 위상이 무작위
- 결과: 풍경 전체에서 "여기저기서 한 줄씩 강조되는" 합창 효과

## 6. 비주얼 스펙

### 6.1. 컨트롤 버튼 (알약)
- `padding: 9px 12px`
- `border-radius: 999px` (완전한 알약)
- `border: 1px solid currentColor`
- `font-size: 11px`
- `letter-spacing: 0.1em`
- 비활성: 투명 배경, 보더만
- 활성(on): 배경이 `currentColor`, 텍스트는 배경색(`--bg`)
- 행 내 `flex: 1`로 균등 분포, `gap: 6px`

### 6.2. 슬라이더
- 트랙: 1px 가로선, `currentColor` 40% 투명
- 손잡이: 16×16px 원, `--text` 색상
- 숫자 표시: 슬라이더 오른쪽 32px 너비

### 6.3. 헤더·라벨
- 헤더 텍스트: 11px, letter-spacing 0.15em, opacity 0.7
- 컨트롤 라벨: 11px, letter-spacing 0.15em, opacity 0.65, UPPERCASE
- 모든 영문 라벨 UPPERCASE 약어 (THIN/LGHT/RGLR/SEMI/BOLD/BLAK 등)

### 6.4. 미리보기 영역
- 흰색 배경, 1px 보더 `currentColor`
- 최소 높이 260px
- 텍스트는 가운데 정렬, `contenteditable="true"`로 직접 편집 가능
- 텍스트 변경 시 부드러운 transition (0.25s)

### 6.5. 모션 원칙
- 모든 상태 변화: `transition: 0.25s` (transform, font-size, color)
- 팔레트 전환: `transition: 0.4s` (background, color)
- 그래픽 등장·소멸: `opacity 0.5s`
- 그래픽 자체 모션: 느리고 절제됨 (3~14초 주기)
- **줄 펄스 (베이스라인):** 줄 단위 `transition: font-weight 0.9s ease-in-out`, 액티브 유지 시간 2.4초/줄. 5.9 참조.

## 7. 핵심 디자인 결정 + 근거

### 7.1. "감정 슬라이더"가 아닌 "타이포그래피 변수"

초기엔 무게/속도/온도 같은 추상 감정 축을 고려했으나, GT Mechanik 패턴(폰트/굵기/기울기 등 구체 변수)으로 결정. 이유:
- 발화자가 자기 감정을 의식적으로 해석하지 않아도 됨
- "만지작거리며 발견"하는 탐색적 UX가 DOCK 문서의 의도와 부합
- 구현·이해 모두 더 명확

### 7.2. FONT(패밀리)와 TONE(장체-평체)을 분리

GT Mechanik의 TONE은 폰트 내적 변형(MONO/SEMI/POLY 폴리포니)이지만, DOCK은 폰트 패밀리 자체가 발화의 음색을 결정해야 함. 그래서 FONT(패밀리)와 TONE(폭 비율)을 별개 축으로 분리.

### 7.3. STAR·COLOR만 랜덤

다른 컨트롤은 모두 발화자의 명시적 결정. STAR(그래픽)와 COLOR(색상)만 우연성에 맡김. 이유:
- 그래픽은 "한국 폰트 specimen 미학"의 핵심이라 잘 못 고르면 안 좋은데, 발화자가 모든 그래픽을 비교 검토하는 건 부담
- 색상은 큐레이팅된 풀에서 우연히 좋은 조합 만나는 것이 디자인의 즐거움

### 7.4. 한글 가변폰트 한계 수용

`wdth`와 `slnt` 축이 있는 한글 가변폰트가 사실상 없음 → CSS `transform`으로 흉내. 자형 일그러짐을 감수. 추후 본인 한글 가변폰트를 디자인하면 교체 가능.

### 7.5. 줄 펄스를 컨트롤로 만들지 않음

사용자가 끄거나 조절할 수 있게 만드는 것도 가능했지만, 베이스라인으로 결정. 이유:
- 줄 펄스는 외벽 디스플레이의 *시스템 특성*이지 발화자의 선택사항이 아님
- "내 메시지가 외벽에서 어떻게 보일지"는 시스템이 정함 (도킹의 의례성)
- 컨트롤이 늘면 만지작거리기 UX가 산만해짐
- WGHT 컨트롤이 펄스 진폭을 *간접적으로* 제어하므로 사용자 결정권은 유지됨

## 8. 기술 제약·고려사항

- **PWA로 작동** (NFC 트리거로 자동 실행 가능해야 함)
- **CDN으로 폰트 로드:**
  - `cdn.jsdelivr.net/gh/orioncactus/pretendard@1.3.9/dist/web/variable/pretendardvariable.min.css`
  - `cdn.jsdelivr.net/npm/@fontsource-variable/noto-serif-kr@5.1.0/index.min.css`
  - `cdn.jsdelivr.net/npm/@fontsource/d2coding@5.0.13/index.css`
- **오프라인 폴백:** 폰트 미로드 시 시스템 폰트로 폴백 (`-apple-system`, `serif`, `monospace`)
- **터치 친화적:** 모든 인터랙티브 요소 최소 36px 높이
- **편집 가능한 미리보기:** `contenteditable="true"`, 60자 카운터 (Phase 04에서 결정된 한도)

## 9. 흐름 맥락 (전후 단계)

**이 화면 진입 직전 (Phase 04 끝):**
- 사용자가 폰을 R2 거치대에 안착시킨 상태
- 60자 메시지 입력 완료
- "다음" 버튼 탭

**이 화면 종료 후 (Phase 06):**
- "맡기기" 탭 → 폰을 4cm 더 깊이 밀어 넣는 2단계 도킹
- 메시지 + 톤 데이터가 서버 전송
- 외벽에 메시지 등장

**즉, 이 화면의 종료 액션이 "다음" 또는 "맡기기"가 되어야 함.** 현재 프로토타입엔 이 종료 액션이 없음 — 추가 필요 (별도 작업).
